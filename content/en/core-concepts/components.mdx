Doc components express semantics such as notes, diffs, structured blocks, and columns. Most components can be used as MDX tags or Markdown directives.

If you mainly need a syntax cheat sheet instead of rendered examples, go straight to [Authoring Syntax](/en/core-concepts/authoring-syntax).

## Callout

Callouts should reflect intent:

- `Tip/Note`: neutral white block for light hints or clarifications.
- `Info`: blue block for stable info or consistency notes.
- `Warning`: orange block for risks or hard constraints.
- `Deprecated`: red block for deprecations and migration notices.

<Tip title="Tip">
Use for hints and small reminders.
</Tip>

<Info title="Info">
Use for stable rules or consistency notes.
</Info>

<Warning title="Warning">
Use for risk or hard constraints.
</Warning>

<Deprecated title="Deprecated">
Use to mark deprecated features or migration notices.
</Deprecated>

## Alerts (GitHub syntax)

Use GitHub-style alerts in Markdown and they will map to callouts automatically.

> [!NOTE]
> This maps to the Note callout and keeps migration syntax consistent.

> [!WARNING]
> Use for risks or hard constraints.

## Math Equations

Inline math uses `$...$`. Block math uses `$$...$$` and renders with KaTeX.

Inline example: $E = mc^2$.

$$
\int_0^1 x^2 \, dx = \frac{1}{3}
$$

## Diff Block

Use to compare two approaches side by side. `beforeLabel` and `afterLabel` are optional when you want the top labels.

    <p>Cache strategy maintained by hand.</p>
  </DiffColumn>
  <DiffColumn highlight>

</DiffBlock>

## Code Block

Use standard Markdown code fences. The renderer adds the shared code styling and copy action.

```ts
export const navStrategy = "filesystem";
export const contentPath = "content";
```

## Colored Diffs

Lines starting with `+` or `-` in code blocks are rendered with diff backgrounds for upgrade guides.

```diff
- cache: "manual"
+ cache: "filesystem"
```

## Code Tabs

Switch between multiple snippets inside one block. Use the `code-group` directive without extra JSX.

::: code-group
```javascript [Chat Completions] subtitle="Structured Outputs"
const completion = await openai.chat.completions.create({
  model: "gpt-5",
  messages: [{ role: "user", content: "Jane, 54 years old" }],
});
```

```javascript [Responses] subtitle="Structured Outputs"
const response = await openai.responses.create({
  model: "gpt-5",
  input: "Jane, 54 years old",
});
```
:::

## Endpoint / ParamTable / ResponseSchema

Use this set for API reference pages. The goal is to split endpoint header, parameters, and response structure into stable semantic blocks instead of loose prose and ad-hoc tables.

<Endpoint
  method="GET"
  path="/v1/docs/[slug]"
  title="Fetch a doc page"
  summary="Returns registry metadata for a route."
  auth="optional"
  since="0.4.0-rc"
>
  <ParamTable title="Path params">
    <ParamField name="slug" type="string[]" required>
      Route segments for the document.
    </ParamField>
    <ParamField name="lang" type='"cn" | "en"' defaultValue="cn">
      Language code.
    </ParamField>
  </ParamTable>

  <ResponseSchema title="200 Response" code="200" mediaType="application/json">
    <SchemaField name="title" type="string" required description="Page title" />
    <SchemaField name="section" type="string" required description="Section name" />
    <SchemaField name="toc" type="TocItem[]" description="Structured headings">
      <SchemaField name="id" type="string" required description="Anchor id" />
      <SchemaField name="title" type="string" required description="Heading text" />
      <SchemaField name="level" type="number" required description="Heading depth" />
    </SchemaField>
  </ResponseSchema>
</Endpoint>

### Directive syntax

~~~mdx
:::endpoint{method="GET" path="/v1/docs/[slug]" title="Fetch a doc page" summary="Returns registry metadata for a route." auth="optional" since="0.4.0-rc"}
:::param-table[Path params]
::param-field[slug]{type="string[]" required="true" description="Route segments for the document."}
::param-field[lang]{type='"cn" | "en"' default="cn" description="Language code."}
:::

:::response-schema[200 Response]{code="200" mediaType="application/json"}
::schema-field[title]{type="string" required="true" description="Page title"}
::schema-field[section]{type="string" required="true" description="Section name"}
:::schema-field[toc]{type="TocItem[]" description="Structured headings"}
::schema-field[id]{type="string" required="true" description="Anchor id"}
::schema-field[title]{type="string" required="true" description="Heading text"}
::schema-field[level]{type="number" required="true" description="Heading depth"}
:::
:::
:::
~~~

## DoDont

This is useful for best practices, migration rules, and review checklists where "do this / don't do this" is the core message.

<DoDont title="When writing API docs">
  <DoItem title="Keep fields stable">
    Public field names, enum values, and error codes should stay stable over time.
  </DoItem>
  <DoItem title="Document failure modes">
    Include 4xx / 5xx response shapes, not just the happy path.
  </DoItem>
  <DontItem title="Do not hide breaking changes in prose">
    Breaking behavior should have explicit markers instead of being buried in a long paragraph.
  </DontItem>
  <DontItem title="Do not rely on examples alone">
    Example responses are useful, but they do not replace field-level structure.
  </DontItem>
</DoDont>

### Directive syntax

~~~mdx
:::do-dont[When writing API docs]
:::do[Keep fields stable]
Public field names, enum values, and error codes should stay stable over time.
:::

:::dont[Do not rely on examples alone]
Example responses do not replace field-level structure.
:::
:::
~~~

## VersionGate

Use this to make it obvious when a capability was introduced, deprecated, or removed.

<VersionGate
  title="Legacy search index"
  since="0.2.0"
  deprecated="0.4.0"
  removedIn="1.0.0"
  replacement="standard search manifest"
>
  When a feature enters migration mode, do not just say "this may change later". Put the version policy next to the feature itself.
</VersionGate>

### Directive syntax

~~~mdx
:::version-gate[Legacy search index]{since="0.2.0" deprecated="0.4.0" removedIn="1.0.0" replacement="standard search manifest"}
When a feature enters migration mode, do not just say "this may change later". Put the version policy next to the feature itself.
:::
~~~
